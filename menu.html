What are dependencies?

When we talk about dependencies we are referring to whatever an object needs in order to work. These dependencies, even though can be expressed in many ways, are divided into 3 categories: Hierarchy, public references, hidden references.

  • An Hierarchy dependency happens ,for example, in unity’s new UI system. Any UI object has to be a child of either the Canvas object or another UI element in order to work at all. So we can say that the parent/child relation is a way to declare a dependency in unity.
  • A public reference dependency, in a unity component, is any public variable of Gameobject or Component type. This is because we are effectively exposing, especially as it shows in unity’s inspector, what the object needs.
  • Hidden references is what we call when objects that depend on other objects in the scene, without declaring them in public variables. This includes the use of singletons, getting references using FindByName, etc... These sort of dependencies are evil as they hide from the developer the requirements for your objects. Whenever we are designing a scene we should focus on what is important and make our life easier. Having to remember that we need to call an object scoreManager instead of Score or making sure we have initialized a specific singleton object when we enter the scene is boring and error prone.

In Blueprints, we won't use hidden dependencies and rely on you to expose your object's dependencies so we can help as much as we can.
