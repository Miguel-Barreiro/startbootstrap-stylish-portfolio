Why use our tool? No frameworks to implment means we dont impose any architecture, any game, including games already in development, can benefit from using it. -reusability - black box principle By - better 3rd party by creating blueprints Understand how to use a specific plugin from the asset store once, implement it on with a blueprint and then never again do you have to remember the finer details again. Just like a tuturial, the Blueprint will proactively ask for whatever it needs. -less bugs - warns when we have missing dependencies - by making it simpler to expose dependencies we avoid using singletons which work like global variables - whenever we instanciate objects it proactivaly tells the developer what we need to do. - As there is no way to have constructors for unity objects, this is a way to garantee the dependencies of dynamicly created objects -easier time designers - by allowing for the logic side to be defined not in the scene but in a meta way we free the designers from this work - we normally divide the scene into logic patterns ( the key to that door, the lich and his 2 minions) so it is easier to manage the scene by allowing to disable them and have them be easier to select and - easier testing - dividing to conquer. By testing each component separately into their own scene we are more confident and allow to easier error spoting of mistakes whenever we make changes. - by having dependencies exposed we can mock the objects attach to them so we can narrow down where the problems can be -No change to the games runtime - it works on the editor. There is no perfomance loss by using it, in fact we are implementing a pattern pool to increase perfomance in some cases

What are dependencies?

When we talk about dependencies we are referring to whatever an object needs in order to work. These dependencies, even though can be expressed in many ways, are divided into 3 categories: Hierarchy, public references, hidden references.

  • An Hierarchy dependency happens ,for example, in unity’s new UI system. Any UI object has to be a child of either the Canvas object or another UI element in order to work at all. So we can say that the parent/child relation is a way to declare a dependency in unity.
  • A public reference dependency, in a unity component, is any public variable of Gameobject or Component type. This is because we are effectively exposing, especially as it shows in unity’s inspector, what the object needs.
  • Hidden references is what we call when objects that depend on other objects in the scene, without declaring them in public variables. This includes the use of singletons, getting references using FindByName, etc... These sort of dependencies are evil as they hide from the developer the requirements for your objects. Whenever we are designing a scene we should focus on what is important and make our life easier. Having to remember that we need to call an object scoreManager instead of Score or making sure we have initialized a specific singleton object when we enter the scene is boring and error prone.

In Blueprints, we won't use hidden dependencies and rely on you to expose your object's dependencies so we can help as much as we can.
